# 5. C++新特性（右值引用与移动语义）
## 右值引用主要用于绑定到即将被销毁的临时对象（右值）。结合移动语义，它可以将资源的所有权从一个对象直接“偷”过来，从而避免深拷贝带来的巨大开销 。结合项目： 在我的网关项目中，当状态机解析出一个完整的数据包（std::vector<char>），需要将其放进对应客户端的发送队列 send_queue 时，如果使用传统的拷贝，会产生一次不必要的内存分配和复制。我使用了移动语义（例如 std::move），直接将数据转移到队列中，极大地降低了高频转发下的内存和CPU开销 。
# 6. 智能指针与循环引用
## 在并发网络模型中，管理连接对象的生命周期极易出错。我在服务端使用了 std::shared_ptr<ClientConnection> 来管理每个客户端的连接状态，并在哈希表中维护这些映射 。当客户端断开连接被移除出哈希表后，引用计数归零，对应的资源就会被安全、自动地析构回收，有效防止了内存泄漏 。关于循环引用，通常发生在两个对象互相持有对方的 shared_ptr 时。解决办法是将其中一方改为 std::weak_ptr。weak_ptr 只观察资源，不增加引用计数 。当需要操作资源时，必须调用 weak_ptr::lock() 尝试将其提升为 shared_ptr，判断指针非空后方可安全访问 。
# 7. epoll机制（LT与ET对比）
## LT（水平触发）模式下，只要缓冲区里还有数据可读，内核就会不断唤醒 epoll_wait；而 ET（边缘触发）模式下，只有当 socket 状态发生变化（如有新数据到达）时，内核才会通知一次 。结合项目： 我的网关追求极致并发，所以采用了“ET模式 + 非阻塞I/O”。因为ET只通知一次，我必须在一个 while(true) 循环中不断调用 recv，直到把缓冲区读空返回 EAGAIN 。如果配合阻塞I/O，最后一次 recv 必然会卡死主线程。非阻塞I/O契合了ET模式，显著减少了系统调用次数，提升了整体吞吐量 。
# 8. TCP协议栈（四次挥手与TIME_WAIT）
## TCP 四次挥手的过程是：主动关闭方发送 FIN 包，被动方回复 ACK 包，进入半关闭状态；被动方处理完剩余业务数据后，也发送 FIN 包，主动方收到后回复 ACK 包，连接彻底释放 。如果在服务端通过 netstat 看到大量 TIME_WAIT 状态，通常是因为服务端主动断开了大量连接（比如踢掉心跳超时的异常设备）引起的 。主动关闭方在发送最后一个 ACK 后，必须等待 2MSL（报文最大生存时间），以防最后一个 ACK 丢失导致对方重传 FIN 。这是正常的 TCP 机制。 
# 9. Linux系统调用（EAGAIN的含义）
## 在非阻塞模式下，调用 recv() 返回 -1 且 errno 为 EAGAIN 或 EWOULDBLOCK 时，这并不代表发生了真正的网络错误，而是表示当前内核接收缓冲区已经为空，没有更多数据可读了 。在我的 ET 事件循环中，遇到这个错误码就意味着本次触发的数据已读完，程序会主动跳出 recv 循环，回到主循环继续监听下一次事件 。
# 10. Reactor模型
## 我的网关核心接入模块采用的是经典的单线程 Reactor 模型 。主线程通过 epoll_wait 作为一个巨大的事件循环。当监听 socket 触发读事件时，Reactor 负责去 accept 接收新连接并将其注册到 epoll 中；当已连接的普通 socket 触发读写事件时，Reactor 会将事件分发给对应的客户端对象，去执行状态机拆包或转发逻辑 。因为网关的性能瓶颈主要在网络 I/O 密集型操作而非 CPU 计算，单线程配合非阻塞 I/O 已经足以打满网卡带宽，同时也省去了多线程编程中复杂的锁同步开销。
# 11. 粘包与半包处理（状态机拆包）
## 针对流式传输，我设计了 12 字节紧凑型的二进制包头：包含 4 字节的魔数（Magic Number）、4 字节的长度、2 字节的类型和 2 字节的设备 ID 。在接收端，我维护了一个动态的接收缓冲区（std::vector<char>）。状态机主要分两个状态：等待包头和等待包体。数据到达后先追加到缓冲区；如果处于“等包头”状态，且缓冲区大于 12 字节，就解析出包体长度并切换到“等包体”状态；当缓冲区未读数据大于等于预期包体长度时，截取出一个完整的数据帧交给上层业务转发，并将状态切回等包头 。这套拼接逻辑解决了粘包和半包问题。
# 12. 路由分发机制
## 为了实现设备端到多个监控端的高效分发，我在内存中使用哈希表（如 unordered_map<int, unordered_set<int>>）来维护订阅关系 。哈希表的键（Key）是采集设备的 DevID，值（Value）是一个集合，存储了所有订阅了该设备的客户端 Socket FD 。当解析出视频流数据时，我通过 O(1) 的时间复杂度查找到所有的订阅者，然后遍历集合，将数据包放入每个订阅者的发送队列中等待发送 。
# 13. 降低空转开销（按需监听EPOLLOUT）
## 这是一个非常关键的性能优化点。绝大部分时间里，TCP的内核发送缓冲区都是不满的，即处于“可写”状态。如果在 epoll 中一直监听 EPOLLOUT 事件，epoll_wait 会被无限次频繁唤醒，导致 CPU 被白白打满（即忙轮询） 。我的策略是：默认只监听读事件。只有当应用层的发送队列有数据，且由于内核缓冲区满了导致发不出去（返回 EAGAIN）时，我才会通过 epoll_ctl 临时开启 EPOLLOUT 监听。内核一旦有空间触发可写事件，我就把数据刷出去。队列清空后，立刻关闭 EPOLLOUT 监听。这样做极大地降低了系统空转开销 。
# 14. 保活与清理机制
## 应用层心跳相较于 TCP 内核底层的 Keep-Alive 机制有明显的优势。TCP Keep-Alive 只能证明底层的网络链路是连通的，但如果对端的业务程序发生了死锁、崩溃或者阻塞卡死，TCP 连接可能依然存在，但实际上设备已经掉线了 。而应用层的心跳包必须由对端的业务代码主动构造并按时发送，能够真实、准确地反映出对端业务进程的存活状态 。
# 15. 慢客户端处理
## 所谓慢客户端，是指由于网络极差或性能拉垮，接收数据非常缓慢的设备。如果不加控制，我发给它的数据会大量积压在服务端的发送队列中，最终引发服务端内存溢出（OOM）崩溃 。我的防御机制是：为每个客户端的发送队列设置一个最大的内存积压阈值（比如 4MB）。当服务端要把新数据压入该客户端的队列时，如果发现队列总大小超过了阈值，网关就会判定其为“慢客户端”，为了自保，服务端会主动断开并清理该连接，避免它拖累整个主循环 。
# 16. Qt异步通信
## 在 Qt 框架中，如果在非主线程（如网络通信子线程）中直接更新 UI 控件，程序会立刻崩溃 。为了安全传递数据，我采用了 Qt 核心的信号与槽（Signals and Slots）机制。底层网络线程在解析出完整的视频帧数据后，通过 emit 发送信号。因为对象处于不同线程，Qt 底层会自动使用队列连接（QueuedConnection）模式，将数据打包放入主 UI 线程的事件循环中。主线程在安全的环境下接管数据并进行界面渲染，彻底避免了多线程并发导致的数据竞争和 UI 假死问题 。
# 17. Bug排查能力
## 项目中遇到过一个偶发的隐蔽 Bug：在极端网络压测下，服务端的内存会持续缓慢上涨。起初我以为是智能指针发生了内存泄漏。后来通过增加关键节点的日志并使用 GDB 观察，发现问题出在异常连接上：某些设备发送了部分半包数据后就断网了。这些不完整的垃圾字节流一直滞留在该连接的接收缓冲区 vector 中。由于死连接越来越多，积压的半包数据吃光了内存。定位后，我引入了定时心跳超时机制，并对缓冲区已消费的前缀定期执行 erase 压缩，彻底解决了内存异常增长的问题 。
# 18. GDB调试实战
## 发生 Core Dump（段错误）时，系统会生成一个 core 文件。我会在终端输入 gdb 可执行程序名 core 进行挂载调试 。进入 GDB 后，我首先会输入 bt（backtrace）命令打印完整的函数调用栈，这能让我瞬间看到崩溃发生在哪一层函数 。然后使用 frame 命令配合栈帧编号（或 up/down）切入到崩溃的上下文中 。接着使用 p（print）指令打印引发崩溃的指针或变量的当前值，排查是否出现了空指针解引用或者数组越界操作 。
