简历中提到你熟练掌握 C++11/14 核心新特性。在你的实际开发中，std::unique_ptr 和 std::shared_ptr 的使用场景有什么区别？有没有遇到过 shared_ptr 循环引用的问题，你是如何解决的？

  Unique指针的使用场景通常是需要独占某一个资源。Shared指针的使用场景通常是多个使用者使用同一资源。关于shared循环引用问题，可以使用weak指针来解决，因为weak指针不会增加引用计数，使用weak指针如果想要访问资源可以通过lock方法，来申请一个临时是shared指针，且申请指针时还会查看对应资源的引用计数是否为0。如果为0表示该资源已经被释放，不可申请访问。

你特别提到了“右值引用”。请结合实际代码场景说明一下，右值引用和移动语义（Move Semantics）是如何帮助程序减少拷贝开销、优化性能的？

  右值引用和移动语义的核心价值，是把“资源所有权转移”从深拷贝变成浅转移。这个项目里真正影响性能的点，首先是把整包数据作为连续的 `vector<char>` 传递，而不是按字段反复重组；当前版本在发送队列里还用了值拷贝，后续如果把 `enqueuePacket` 改成接收 `vector<char>&&`，再配合 `std::move` 入队，就能让大包只发生所有权转移，减少一次整包复制。
  所以在我的理解里，移动语义不是为了“把数据变快读出来”，而是为了让网络服务里那些大对象在队列、返回值和容器之间传递时，尽量不做无意义的内存拷贝。

关于 Lambda 表达式，它在你的 C++ 项目中通常扮演什么角色？在使用按引用（[&]）捕获外部变量时，有什么潜在的内存安全隐患（比如悬垂引用）？

  Lambda 在我的项目里更多是用来写一些一次性的回调、条件判断或者事件处理逻辑，代码会更短，也更贴近“在这里定义、在这里使用”的语义。按引用捕获本身没有问题，问题在于它依赖外部变量的生命周期：如果 Lambda 被保存下来延迟执行，而外部局部变量已经销毁，再访问它就会变成悬垂引用。
  所以在网络服务这种异步场景里，我通常会更谨慎地使用 `[&]`，能按值捕获就尽量按值捕获；如果确实要共享对象，就让对象进入智能指针管理范围，这样生命周期更清晰，也更安全。

针对 C++ 的内存管理，在长期运行的网关服务端程序中，如果频繁地 new 和 delete 会导致什么后果？你平时有了解或使用过内存池技术吗？

  不好意思，这个我没有了解过，但我认为频繁的new和delete可能会导致内存中出现大量的碎片，降低内存使用效率。额，内存池我其实没有了解过。
  频繁的 new/delete 会在用户态和内核态之间产生大量的系统调用切换（底层调用 malloc/free，引发 brk 或 mmap），同时会在堆上产生大量内存碎片，降低分配效率。内存池技术就是预先向系统申请一大块连续内存，在应用层自己进行切分和管理。针对固定大小的报文对象，使用对象池可以做到 O(1) 的分配速度，极大地提升网关的高并发性能。

你提到深入理解 Linux 操作系统原理。那么当你的 C++ 程序调用一次 send() 将数据发送到网络时，数据在用户态和内核态之间经历了怎样的流转和拷贝过程？

  `send()` 是一个系统调用，调用时会把用户态缓冲区里的数据拷贝到内核态的 socket 发送缓冲区。真正发到网卡的过程由内核协议栈异步完成，所以 `send()` 返回的只是“这次成功写入内核缓冲区的字节数”，不等于已经发到对端。
  在我的项目里，`flushPendingWrites()` 就是围绕这个行为写的：先尽量把数据写进内核，如果内核发送缓冲区满了返回 `EAGAIN`，我就保留剩余数据并依赖 `EPOLLOUT` 下次再发；同时还加了 `MSG_NOSIGNAL`，避免对端关闭时触发 `SIGPIPE` 把整个进程打掉。

熟练掌握 TCP/IP 协议栈：请问为什么建立连接需要“三次握手”，而断开连接却需要“四次挥手”？

  三次握手的关键点是：客户端发起连接、服务端确认收到、客户端再确认一次。第三次握手不是为了“多余确认”，而是为了让双方都确认收发路径是通的，避免半开连接。四次挥手则是因为 TCP 是全双工的，两个方向的关闭是独立的，所以需要分别发送 FIN 和 ACK。
  你可以把它理解成：建立连接时要确认“双方都能说话”，断开连接时要分别确认“双方都不再说话了”，这就是握手和挥手次数不同的根本原因。

在 TCP 四次挥手过程中，TIME_WAIT 状态存在的意义是什么？如果你的 Linux 服务器上出现了大量的 TIME_WAIT 状态，通常是什么原因导致的？该如何优化配置？

  TIME_WAIT 的意义主要有两个：一是确保最后一个 ACK 有机会被对方收到，二是让网络里延迟到达的旧报文自然失效，避免它们污染下一次连接。这个状态通常由主动关闭连接的一方进入，持续 2MSL。
  在我的服务端里，`setsockopt(..., SO_REUSEADDR, ...)` 已经打开了，这样服务热重启时可以更顺利地复用本地端口；如果业务本身大量主动断开短连接，才需要进一步从连接复用、长连接化或者架构层面去减少 TIME_WAIT 的积累。

针对高并发网络编程，你深入掌握了 epoll 机制。请问 epoll 相比传统的 select 和 poll，解决的最核心性能痛点是什么？

  `epoll` 解决的核心痛点，是避免 `select/poll` 每次都把所有 fd 重新扫描一遍。`select/poll` 的开销会随着监听数量线性增长，而 `epoll` 把关注的 fd 维护在内核里，调用 `epoll_wait()` 时只返回真正就绪的事件，性能会更稳定。
  在我的这个服务端里，连接数一多以后，读写事件本身才是热点，所以我选择 `epoll + 非阻塞 IO + ET`，把“谁就绪了”这件事交给内核，把“怎么处理就绪事件”留给应用层状态机。

epoll 的 LT（水平触发）和 ET（边缘触发）模式有什么本质区别？你在 IIoT 网关项目中主要使用的是哪种模式？为什么做此选择？

  LT 的特点是“只要条件还满足就会反复通知”，ET 的特点是“状态从不就绪变成就绪时只通知一次”。我在这个项目里用的是 ET，因为它更适合高频读写场景，能减少重复唤醒和无效系统调用。
  但 ET 也要求代码更严格，所以我在接收端写了“读到 `EAGAIN` 为止”的循环，在发送端写了“发不完就挂 `EPOLLOUT`”的逻辑，保证不会因为少读一次、少写一次而丢事件。

在这个项目中，你基于 epoll 和非阻塞 IO 实现了 Reactor 模型。请详细描述一下你的 Reactor 架构设计——是单 Reactor 单线程，还是多 Reactor 多线程模型？

  我在项目中采用的是单 Reactor 单线程模型，整个程序只有一个主线程，没有额外创建工作线程。
  这个选择和当前业务比较匹配：接收、拆包、路由、发送都放在同一个事件循环里，逻辑简单、状态一致，也没有多线程锁竞争的问题。对这个视频流网关来说，先把 IO 路径跑稳比一开始就上复杂并发模型更合适。

简历中提到“通过 EPOLLOUT 按需唤醒降低空转开销”。请问在什么具体的缓冲区状态下你需要主动去注册监听 EPOLLOUT 事件？什么时候又必须将其移除？

  在我的代码里，只有当某个连接的发送队列里真的还有待发送数据时，我才会去开启 `EPOLLOUT`。具体来说，先把新包放进 `send_queue`，再尝试 `flushPendingWrites()`；如果一次没发完、内核缓冲区也暂时写不进去，就让 `EPOLLOUT` 继续帮我唤醒。
  相反，只要 `send_queue` 已经空了，我就立刻把 `EPOLLOUT` 去掉。这样做的目的很明确：避免一个本来空闲的连接因为可写事件一直被唤醒，造成 CPU 空转。

项目中你设计了“自定义二进制协议与状态机拆包逻辑”。能否详细说明一下你的协议报文头（Header）里包含了哪些关键字段？

  这个协议头在 `protocol.h` 里定义得很清楚，固定是 12 字节：`magic`、`length`、`type`、`devId`。其中 `magic` 用来快速校验是不是我这个协议的包，`length` 表示包体长度，`type` 用来区分心跳、视频流和订阅请求，`devId` 用来标识是哪台设备发来的数据。
  这样设计的好处是拆包时逻辑非常直接：先看魔法数和长度是否合法，再根据类型决定是转发、订阅还是保活，既能防止脏数据误入，也方便后续扩展新的消息类型。

针对 TCP 字节流传输固有的“粘包”和“半包”问题，你的状态机具体是如何配合缓冲队列进行截断和拼接处理的？

  我在 `ClientConnection::extractPacket()` 里做的是一个显式状态机：先等够包头，再根据包头里的 `length` 去等包体。`recv_buffer` 负责暂存所有收到的数据，`read_offset` 负责记录当前消费到哪儿了，所以即使一次 `recv()` 里拿到多个包，也可以连续拆；如果只拿到半包，就先返回，等下次数据再继续拼。
  如果包头校验失败，比如 `magic` 不对或者长度超限，我会向前挪一个字节重新对齐，避免错误包头把后面的数据全部带偏。这一层处理就是专门为 TCP 的粘包和半包准备的。

你的协议还要处理工业设备的“视频流链路解析”。视频流数据量通常较大，你在接收端有做零拷贝或者特殊的内存管理优化来防止内存频繁分配吗？

  这个版本还没有做真正意义上的零拷贝。当前流程是：先 `recv` 到临时缓冲区，再追加到连接自己的 `recv_buffer`，拆出来的完整包再放进发送队列，所以中间仍然有必要的内存拷贝。
  但在内存管理上我已经尽量做到 RAII 化了：连接对象用 `shared_ptr<ClientConnection>` 管理，收发缓冲都放在 `vector` 和 `deque` 里，连接断开时由统一的清理逻辑释放资源。后续如果继续优化性能，可以考虑环形缓冲区、`writev` 或更激进的零拷贝方案。

简历中写道你增加了“心跳保活、超时清理”机制。面对成千上万个并发连接，你是如何在服务端高效地找出哪些连接已经超时的？（比如是否使用了时间轮、定时器小根堆等数据结构？）

  额，这个暂时不知道
  面对大规模并发连接，常规的轮询遍历开销太大。我通常会考虑引入**时间轮（Time Wheel）**算法，或者使用基于最小堆的定时器（如 C++ 的 std::priority_queue）。将连接的超时时间作为 Key，每次只需检查堆顶元素是否过期即可，复杂度降至 O(1) 或 O(logN)。在 Linux 环境下，还可以结合 timerfd 将定时器事件统一交给 epoll 管理。

你提到了“构建设备订阅路由与发送队列机制”。当遇到慢客户端（即某个监控端消费数据的速度远低于底层设备发送的速度）时，你的发送队列是如何避免无限堆积导致 OOM 的？

  这个项目里我是直接给每个连接设置了发送队列上限，也就是 `max_pending_bytes_per_client`。`ClientConnection` 里有 `queued_bytes` 统计当前积压的数据量，`enqueuePacket()` 会在超过阈值时直接返回失败，主循环随后把这个慢客户端踢掉，避免它无限占用内存。
  这套策略的思路很简单：服务端不能为了照顾一个消费太慢的订阅者，拖垮整条转发链路。对实时视频流来说，保住系统整体稳定性比死等一个慢客户端更重要。

作为 C++ 服务端研发，你配套开发了轻量级 Qt5 客户端。你觉得 Qt 的“信号与槽”机制底层是如何解耦对象的？它和 C++ 标准库中的 std::function 回调有什么异同？

  Ui界面的组件只需要发送相对应的信号，对应的槽函数接到信号以后就去执行并返回结果，组件不需要去担心函数具体是如何实现的。与回调类似的一点是，槽函数也是传入参数，执行并返回结果，本质是一个函数，但不同的是，槽函数的解耦性更高，当ui界面发生变化时，只需要修稿connect函数的连接关系就行，不用修改槽函数内部的逻辑
