# 1. 为什么选择单线程 Reactor？何时会成为瓶颈？

## 回答：因为网关的核心任务是纯粹的网络数据流收发，属于典型的 I/O 密集型场景。在千兆网卡以下，单线程配合 epoll 非阻塞 I/O 已经能轻松跑满带宽，同时彻底避免了多线程抢占资源带来的锁竞争开销和复杂的上下文切换。

## 瓶颈点：当业务不仅是转发，还需要进行重度 CPU 计算（比如视频流的软解码、高强度的 TLS/SSL 报文加解密）时，单线程的主循环就会被计算阻塞，导致网络事件处理延迟，这时候就需要引入多线程（如 Reactor 主从模型或业务线程池）。

# 2. epoll 的 ET 模式与非阻塞 I/O

## 回答：水平触发（LT）只要缓冲区有数据就会一直唤醒，而边缘触发（ET）只有在状态跳变（如新数据到达）时才通知一次。

## 必须非阻塞的原因：因为 ET 只通知一次，我必须在一个 while 循环里不断调用 recv 把当前缓冲区的数据全部读干，直到读出 EAGAIN。如果用阻塞 I/O，最后一次 recv 会由于没有数据可读而死锁，直接卡死整个 Reactor 主循环。

# 3. 协议状态机与 OOM 防御

## 回答：包头设计了 12 字节（魔数+长度+类型+设备ID）。我的状态机有两个核心状态：等待包头 和 等待包体。

## OOM 防御：在解析出长度字段后，我会立即将该长度与预设的常量最大阈值（如 kMaxBodyBytes 16MB）进行校验。如果超出，直接判定为恶意注入或脏数据，丢弃当前缓冲区数据甚至主动断开连接，防止 vector 盲目扩容导致 OOM。

# 4. 魔法数校验失败与重连后果

## 回答：一旦魔法数校验失败，说明 TCP 字节流已经错位或被污染。由于 TCP 是流式协议，很难精准找回下一个正确的包头边界，为了防止后续解析全部崩溃，我会直接调用 removeClient 主动断开并清理该连接。

## 重连后果：在工业环境下，大量设备频繁断开重连会引发类似 SYN Flood 的雪崩效应，导致服务端瞬间消耗大量 CPU 去处理三次握手和对象分配，甚至可能耗尽文件描述符（FD）。

# 5. 拦截 SIGPIPE 信号

## 回答：在 Linux 系统中，如果往一个对端已经关闭的 Socket 调用 send，内核会默认向当前进程发送 SIGPIPE 信号。该信号的默认行为是直接终止整个进程。我在 send 时加上 MSG_NOSIGNAL 标志位，可以抑制这个信号的产生，只让 send 返回 -1 并置错 EPIPE，从而优雅地清理断开的连接，保证网关 7x24 小时运行。

# 6. 极致的资源开销优化（6MB，3% CPU）

## 回答：主要得益于两点：

## 内存复用与压缩：接收缓冲区使用了连续内存的数据结构，通过记录 read_offset 实现前缀压缩（达到一定阈值后才批量 erase 废弃的头部字节），避免了每次收包都频繁 malloc/free 带来的内存碎片。

## 避免空转轮询：我采用了按需唤醒策略。平时绝不监听 EPOLLOUT，只有当应用层发送队列出现积压，且内核发送缓冲区已满时，才通过 epoll_ctl 开启可写监听，数据发完立即关闭。这极大降低了 CPU 占用。

# 7. Qt 跨线程通信

## 回答：Qt 底层通过**信号与槽（Signals and Slots）**机制结合事件循环保证线程安全。当底层网络子线程 emit 一个信号时，由于接收者（UI 界面）在主线程，Qt 会自动采用队列连接（QueuedConnection）。它将信号及携带的数据打包成一个事件，投递到主线程的事件队列中。主线程在下一次事件循环时安全地取出并执行，彻底解耦了多线程的数据竞争。

二、 C++ 与系统底层技术
# 8. 右值引用与移动语义

## 回答：移动语义可以将对象内部资源的控制权直接“偷走”，避免深拷贝。在网关中，当我从缓冲区成功提取出一个完整的包（如 vector<char> packet），需要将其推入对应客户端的待发送队列时，我会使用 std::move(packet)。这直接转移了内存指针，将 O(N) 的数据拷贝开销降为了 O(1) 的指针赋值。

# 9. 智能指针与循环引用

## 回答：在维护设备订阅关系时，使用 std::shared_ptr 管理客户端对象。为了防止互相引用导致引用计数永远无法清零（内存泄漏），对于仅需要“观察”而不控制生命周期的场景，必须使用 std::weak_ptr。使用前调用 lock() 尝试将其提升为 shared_ptr，如果返回不为空，说明连接存活，可安全操作。

# 10. 内存泄漏排查

## 回答：我首选 Valgrind 的 Memcheck 工具。在编译时加上 -g 选项保留调试信息，然后用 Valgrind 运行程序。当进程结束时，它会输出详细的报告，包括“绝对丢失（definitely lost）”的字节数，并精准定位到发生泄漏的代码行（如未被释放的 new 或 malloc）。如果是排查运行中的偶发泄漏，我也会配合 GDB 观察内存驻留情况。

# 11. TCP 四次挥手与 TIME_WAIT

## 回答：主动关闭方发送 FIN，被动方回 ACK；接着被动方发 FIN，主动方回 ACK。

## TIME_WAIT 优化：主动关闭连接的一方必定会进入 TIME_WAIT 状态并维持 2MSL。服务端出现大量该状态，通常是因为服务端主动剔除了大量超时/异常客户端。为了防止端口耗尽，可以通过修改内核参数 sysctl，开启 net.ipv4.tcp_tw_reuse = 1 允许复用 TIME_WAIT 的套接字，但要慎重使用。

# 12. GDB 调试 Core Dump

## 回答：

## 运行 gdb <可执行文件> <core文件>。

## 输入 bt（backtrace）查看崩溃时的调用栈，锁定崩溃发生的函数。

## 使用 frame <编号> 切入对应的栈帧。

## 使用 print 或 p 打印关键指针变量的值，检查是否是空指针解引用、越界访问或被破坏的魔法数导致了段错误。

# 13. CMake 交叉编译

## 回答：我会单独编写一个 Toolchain.cmake 文件。在其中配置 CMAKE_SYSTEM_NAME（如 Linux）和 CMAKE_SYSTEM_PROCESSOR（如 ARM）。最关键的是重定向编译器路径，设置 CMAKE_C_COMPILER 和 CMAKE_CXX_COMPILER 指向目标平台的交叉编译工具链（例如 aarch64-linux-gnu-gcc）。构建时通过 -DCMAKE_TOOLCHAIN_FILE 传入该配置文件即可。

三、 计算机基础与综合能力
# 14. 哈希表规模达百万级（十万级以上）

## 回答：std::unordered_map 在达到数百万级别时，随着负载因子升高会发生剧烈的 Rehashing（重新分配桶并全量复制），在此期间整个网关会因单线程被卡死而停顿。哈希冲突加剧也会让查找性能从 O(1) 退化为链表遍历。

## 应对方案：在单机内存中，我会考虑引入分段锁的并发哈希表或跳表（SkipList）。如果规模实在太大，应该把状态存储剥离到内存数据库（如 Redis）中，进行分布式管理。

# 15. 分布式网关扩展（状态同步与消息路由）

## 回答：

## 状态同步：单机内存的哈希表不再适用。我会引入 Redis 集群，设备上线时向 Redis 注册路由信息（如 Device_A -> Node_1）。

## 消息路由：我会引入消息中间件（如 Kafka 或 RabbitMQ）。网关节点收到视频流后，发布到以设备 ID 命名的 Topic 中。订阅端只需连接最近的节点并监听对应 Topic，实现解耦和无限横向扩容。

# 16. 字节流中搜索特征码（12字节包头）算法

## 回答：这是一个经典的字符串/子串匹配问题。暴力匹配是 O(M*N)。为了追求网关的极致性能，我会采用 KMP 算法。因为我们自己定义的 12 字节包头特征码是已知的，可以预先计算出它的 next 数组（部分匹配表）。在扫描连续字节流时，一旦失配，可以直接通过 next 数组跳过无效比对，将时间复杂度降到 O(N)。

# 17. AI 辅助定位疑难 Bug

## 回答：网关曾出现过极端压测下的慢速内存泄漏。我起初查不出哪里没释放。我将相关的状态机逻辑和缓冲区的清理代码脱敏后喂给 AI。AI 指出了一处盲点：当客户端发送了不完整的半包后突然断电，由于没有触发后续事件，这部分“脏字节”会永远滞留在对应的 vector 中。根据这个启发，我立刻在心跳超时逻辑中加入了强制回收游离缓存的逻辑，彻底解决了隐患。

## 回答：遇到突发的高可用死锁问题，我会遵循“先止血，后根治”的原则。
