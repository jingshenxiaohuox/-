## 这个项目的整体架构图你能画一下吗？客户端和服务端是怎么交互的？

    中间是核心，用 C++ 开发，基于 epoll 边缘触发模式实现高并发。它负责两件事：一是接收所有采集端的数据，解析协议后按设备 ID 缓存；二是响应观看端的请求，按需转发数据。

    右边是客户观看端，也是 Qt 开发。它先向服务端请求在线设备列表，用户选择某个摄像头后，服务端就开始实时转发该设备的数据，观看端收到后解码播放。

    交互流程上，采集端和服务端是长连接持续推送数据；观看端和服务端是请求-响应模式，订阅后服务端主动推送。整个系统通过自定义二进制协议保证数据传输的可靠性和完整性。

## 为什么用 epoll 而不是 select/poll？epoll 的优势具体体现在哪里？

    1. 监视的文件描述符数量上限不同

    select 受限于 FD_SETSIZE，默认 1024，虽然可以重新编译内核修改，但不够灵活。

    poll 使用链表管理，理论上没有上限，只受系统资源限制。

    epoll 同样没有固定上限，只受系统最大文件描述符数限制（/proc/sys/fs/file-max）。

    2. 事件通知机制不同

    select/poll 每次调用都要把整个 fd 集合从用户态拷贝到内核态，并且内核需要O(n) 遍历所有 fd 检查事件。

    epoll 在内核中维护一个红黑树管理所有待监视的 fd，同时维护一个就绪链表。当事件发生时，epoll 通过回调机制把就绪的 fd 自动加入到就绪链表中，用户调用 epoll_wait 时只需要直接从就绪链表拷贝到用户态，不需要遍历所有 fd。

    3. 触发模式

    select/poll 只支持水平触发（LT）。

    epoll 额外支持边缘触发（ET），配合非阻塞 I/O 可以进一步减少系统调用次数，提升效率。

    所以在高并发、大连接数的场景下，epoll 的性能远优于 select/poll。

## 你用了 epoll 的 ET 模式，ET 和 LT 的区别是什么？为什么选 ET？
    ET（边缘触发）和 LT（水平触发）的主要区别在于通知机制和编程模型。

    LT（水平触发）：只要文件描述符还有未处理的数据，每次调用 epoll_wait 都会返回这个 fd。编程逻辑简单，不容易漏事件，但可能产生多次无效的系统调用，因为内核会反复通知直到数据读完。

    ET（边缘触发）：只在 fd 状态发生变化时（比如从不可读变为可读）通知一次。之后即使还有数据没读完，也不会再通知，直到下一次状态变化。

    为什么我选 ET：
    在我的工业物联网项目中，设备数量大、数据流量高，我希望尽可能减少 epoll_wait 的返回次数，降低系统调用开销，提升吞吐量。ET 模式配合非阻塞 I/O，在 while 循环里一次性把数据读完（直到返回 EAGAIN），可以最大化性能。
    代价是编程复杂度更高，需要小心处理数据读取，否则容易漏事件。但在高并发场景下，这个代价是值得的。
